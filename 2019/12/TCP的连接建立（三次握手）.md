## 三次握手详解

TCP连接采用`C/S`的方式，客户端主动发起连接建立，服务端被动等待连接建立

1. 开始A(客户端)和B(服务端)都处于`CLOSED`(关闭)状态
2. B创建`TCB`(传输控制块，存储每一个连接中的一些重要信息)，进入`LISTEN`(收听)状态，等待客户端请求
3. A创建`TCB`打算建立TCP连接，向B发送SYN报文段(`SYN = 1`的报文段)，首部中的`SYN = 1`，同时选择一个ISN(初始序号) `seq = x`。TCP规定，**SYN报文段不能携带数据，但要消耗一个序号**。此时A进入`SYN-SENT`(同步已发送)状态
4. B收到A的请求报文段后，如果同意建立连接，则向A发送确认。确认报文中SYN位和ACK位都为1，确认号`ack = x + 1`，同时自己选择一个初始序号`seq = y`。**注意，该报文段也不能携带数据，但同样要消耗一个序号**。此时B进入`SYN-RCVD`(同步收到)状态
5. A收到B的确认后，还要向B给确认。确认报文段的`ACK = 1`，确认号`ack = y + 1`，而自己的序号`seq = x + 1`。**TCP规定，ACK报文可以携带数据，但如果不携带则不消耗序号，这种情况下，下一个报文段的序号依旧是seq = x + 1**。B收到确认后，TCP建立就完成了，双方进入`ESTABLISHED`(已建立连接)状态

以上`3、4、5`三步为三次握手

## 为什么最后A还要向B发送一次确认

假设有这样一种情况，A发出一个连接请求，但是该请求在某些网络结点长时间滞留，导致延误到连接释放以后的某个时间才到达B。本来这是一个已经失效的报文段，但B收到这个报文段以为A又重新发送了一次连接请求，于是B向A发送确认报文段，同意本次连接。如果没有最后一次确认，那么此时连接已经建立了。

由于现在A并没有发出建立连接的请求，因此不会理睬B的确认，也不会向B发送数据。但B却以为已经建立了连接，一直等待A发送数据。B的许多资源就被浪费了

但如果采用三次握手，上述的异常情况下，A不会向B的确认发出确认，B由于收不到确认，就知道A并没有要求建立连接
